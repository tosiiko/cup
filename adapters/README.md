# CUP Adapters

This folder holds the official server-side adapter work for CUP.

## Current State

Production adapters in this repo:

- `python/`
- `go/`

- `node/`
- `typescript/`
- `rust/`
- `java/`
- `dotnet/`
- `php/`
- `ruby/`
- `elixir/`
- `kotlin/`
- `swift/`
- `dart/`
- `clojure/`
- `lua/`
- `zig/`
- `nim/`
- `ocaml/`
- `perl/`
- `haskell/`

Important distinction:

- Python and Go are real implemented adapters today
- Node and TypeScript are alpha wrapper package paths with dedicated helper APIs
- Rust and Java now have alpha source adapters in-repo
- the rest are ecosystem-native scaffolds that establish the official package path and source layout for future implementation

The machine-readable registry now separates:

- namespace intent
- in-repo implementation state
- implementation kind
- maturity
- publication status
- protocol version support

## Official Namespace Policy

CUP uses two naming layers:

- core runtime package:
  `@tosiiko/cup`
- language adapter family:
  `*-cup`

Examples:

- `py-cup`
- `go-cup`
- `rs-cup`
- `java-cup`

These names are tracked in [`namespaces.json`](./namespaces.json) and explained in [`../docs/adapters.md`](../docs/adapters.md).

Important:

- this registry is the project’s canonical naming plan
- it does **not** mean every name is already published on every ecosystem registry
- a name becomes truly reserved only when it is published or claimed in the relevant package registry

## Adapter Rules

An official CUP adapter should:

- emit protocol-compatible `v1` views
- preserve `meta.version`, `meta.lang`, and `meta.generator`
- pass the shared contract tests where applicable
- provide validation helpers
- document starter-safe policy validation
- include examples or starters that show the intended server pattern

## Automation

The scaffold family is generated by:

- [`../scripts/bootstrap-adapters.mjs`](../scripts/bootstrap-adapters.mjs)

That script reads [`namespaces.json`](./namespaces.json) and keeps the adapter family consistent.

## Naming Goal

The long-term goal is to make the adapter family easy to recognize:

- one protocol
- one runtime
- one consistent language adapter namespace
